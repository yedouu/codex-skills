# Codex Tasks Template

## 任务格式

```markdown
## 任务：{标题}

### 目标
说明要实现或验证什么。

### 涉及文件
- path/to/file
- path/to/module

### 实现步骤
1. ...
2. ...
3. ...

### 验证方式
- 单元测试：
- 集成测试：
- 手动验证：
- 回归检查：

### 风险检查点
- license：
- 安全：
- 性能：
- 兼容性：
- 可维护性：

### 不要做的事情
- 不要直接复制外部代码。
- 不要引入未评估 license 的依赖。
- 不要扩大无关改动范围。
- 不要跳过必要验证。
```

## 从调研结论转任务

1. 只把推荐方案和高价值可迁移思想转成任务。
2. 每个任务应能独立交给 Codex 执行。
3. 对可能涉及外部代码的任务，写成“借鉴架构/接口思想”，不要写成“复制实现”。
4. 如果当前项目上下文不足，任务应先要求 Codex 读取相关文件并确认技术栈。
5. 对风险较高的实现，先创建 spike 或验证任务，再创建完整实现任务。

## 任务拆分建议

- 先做最小可验证路径。
- 再接入真实数据或真实 API。
- 再完善错误处理、配置、测试和文档。
- 最后做性能、体验或安全增强。
