# Chapter Template

Use this for every `.learning-roadmap/chapters/*.md` file. Fill it with real project-specific content.

Chapters should read like teaching tutorials, not task checklists. Default order: theory and design first, source-code design interpretation next, simplified teaching strategy next, and implementation practice last.

```markdown
# 第 N 章：标题

## 0. 本章导读

用 2-4 段话说明本章要学什么、为什么重要、它和整个项目主线的关系。不要急着列任务。

## 1. 本章要解决的核心问题

必须回答：
- 这个模块/概念为什么存在？
- 它解决了项目里的什么问题？
- 如果不设计这个模块，系统会有什么缺陷？
- 它和上一章、下一章有什么关系？

## 2. 理论背景与核心概念

这一节重点讲理论，不要急着写代码。

要求：
- 用通俗语言解释概念；
- 给出必要的数学、算法、框架或工程背景；
- 对复杂概念先讲直觉，再讲正式定义；
- 不要直接堆源码；
- 可以使用伪代码、流程图、类比帮助理解。

按重要性调整深度：
- A 级核心内容：理论背景和实现思路要详细，必须解释设计原因；
- B 级支撑内容：讲清楚作用和基本原理，不必过度展开源码细节；
- C 级配置内容：只讲用途、常改字段、常见错误，不安排大量编码任务；
- D 级高级内容：只说明它解决什么高级问题，放到后期章节；
- E 级内容：明确说明暂时跳过原因。

示例：
- 粒子滤波章节先讲概率分布、粒子、权重、运动模型、观测模型、重采样；
- ROS2 节点章节先讲生命周期、订阅发布、回调、状态门控；
- 配置章节只讲用途、常改字段和风险，不深挖构建系统细节。

## 3. 实现思路

这一节讲“怎么设计”，但还不进入具体编码步骤。

必须说明：
- 输入是什么；
- 输出是什么；
- 内部状态是什么；
- 数据如何流动；
- 模块如何协作；
- 哪些复杂度在教学版中先省略；
- 哪些复杂度是原项目必须保留的。

## 4. 原项目设计解读

这一节把理论映射到原项目源码。

必须说明：
- 原项目对应哪些文件；
- 每个文件在设计中承担什么角色；
- 为什么原项目要这样拆模块；
- 哪些地方是算法核心；
- 哪些地方是工程集成；
- 哪些地方初学时可以暂时跳过。

如果某些判断只是推断，标注“根据目录结构推断”。如果需要运行项目确认，标注“需要运行项目确认”。

## 5. 教学版简化策略

说明为了学习，我们如何压缩原项目复杂度。

必须包含：
- 教学版保留什么；
- 教学版删掉什么；
- 为什么可以暂时删掉；
- 这些被删掉的内容后面在哪一章补回来。

## 6. 最小实现方案

这一节才开始说要实现哪些结构。

包括：
- 需要定义哪些类 / 函数 / 状态；
- 每个类或函数的职责；
- 输入输出；
- 推荐文件组织；
- 不要直接给大段完整代码，除非用户要求。

## 7. 推荐编码步骤

把实现拆成小步骤。

要求：
- 每一步都要解释为什么；
- 不要只列步骤；
- 每一步最好能独立验证。

基础任务：
完成最低可运行目标。

进阶任务：
加入真实项目中的关键支撑细节。

挑战任务：
尝试接近原项目结构或进行小步重构。

## 8. 完成标准

完成本章后，你应该能够：
1. 用自己的话解释这个模块/概念为什么存在；
2. 说明它解决了项目中的什么问题；
3. 解释原项目为什么这样设计；
4. 独立设计一个教学版最小实现；
5. 知道哪些复杂度被教学版省略；
6. 知道原项目对应哪些文件；
7. 能运行一个验证用例；
8. 能定位常见错误。

## 9. 验证方式

- 运行命令：
- 操作步骤：
- 预期现象：
- 失败时检查点：
- 调试建议：

## 10. 与原项目的对比

必须说明：
- 教学版和原项目差在哪里；
- 原项目多出来的复杂度是为了解决什么问题；
- 哪些设计值得后续学习；
- 哪些设计现在可以先跳过。

## 11. 自测问题

问题应该偏理解和设计，而不只是记忆。

示例：
- 为什么这里需要状态门控？
- 为什么不能只保留最高权重粒子？
- 为什么配置文件只需要先了解用途而不是深挖？

## 12. 下一章预告

说明下一章如何承接本章。
```
